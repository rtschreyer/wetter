Zielstellung

Implementieren Sie einen Service, der für beliebige Geo-Koordinaten die Wettervorhersage abruft (https://developer.forecast.io/docs/v2#forecast_call). Beachten Sie dabei Stichwor- te wie Performance, Reusability, Code Quality und Robustness.

Schreiben Sie einen Controller innerhalb einer lauffähigen Applikation, der den von Ihnen implementierten Service nutzt und die von der API erhaltenen Daten in geeigneter Form anzeigt.

Fügen Sie optional einen Unit-Test für den Service hinzu.

Fragestellungen

Welche Ausbaumöglichkeiten sehen Sie für den von Ihnen implementierten Service?
Was hier implementiert ist, ist nur eine Beispiel App, ohne wirklichen Nutzwert. Trotzdem sollten z.B. die Konfigurationsparameter in den localstorage ausgelagert werden. Man könnte dafür eine view mit einem Eingabefeld anlegen, in das der user seinen eigenen Apikey eintragen kann.
Es könnten mehrere Sprachen zur Auswahl stehen. Als Dateneingabe könnten andere Online-Services angebunden werden. Jede beliebige Quelle, die spatiale Geodaten ausgibt kann genutzt werden. Dafü benötigt mein Service nur die Argumente 'lon' und 'lat', dass Frontend braucht zusätzlich einen Ortsnamen.
Wie würden Sie den Service dokumentieren, so dass andere Entwickler ihn ohne große Probleme produktiv einsetzen können?
Für Entwickler, die den Sourcecode anpassen wollen, befinden sich bereits Kommentare an dafür vorgesehenen Stellen.
Idealerweise sollte eine ausführliche Dokumentation z.B. als Wiki festgehalten werden. Bei fortlaufender Entwicklung auch als Versionskontrolle mit Changelog.
Beim derzeitigen Umfang des Projektes genügt allerdings bereits eine readme Datei. Gegenenefalls im md Format, wenn es im git repository gepflegt wird.
In welcher Weise ist der zweite verfügbare API-Call (https://developer.forecast.io/doc- s/v2#time_call) einsetzbar?
Z.B. um das Wetter bei bestimmten zurückliegenden Ereignissen heraus zu finden.
Oder um den Verlauf von Temperatur, Bewölkung, Niederschlagsneigung an einem bestimmten Ort über einen längeren Zeitraum zu dokumentieren.
Allerdings müssten dazu viele calls hintereinander abgesetzt werden und die Ergebnisse idealerweise in einer Datenbank gespeichert werden.
Begründen Sie bitte in kurzer Form, warum Sie sich für bestimmte Implementierungsde- tails entschieden haben.
Ganz wichtig war für mich, dass die Konfigurationsparameter Sprache und Apikey möglichst leicht änderbar sein sollten. Der Apikey sollte niemals hardcodiert irgendwo im Sourcecode stehen. Daher habe ich die Möglichkeit implementiert eine externe json-Datei auslesen zu können. Das ist noch nicht ideal gelöst.
Um den Apikey oder die Sprache zu ändern, reicht es die Entsprechenden Einträge in der Datei config.json zu ändern. Der Parameter 'lang' ädert dabei nur die Ausgabesprache des forecast.io sevices. Für eine tatsäliche Globalisierung müsste noch wesentlich mehr geändert werden. Für diesen Zweck und auch, da es eventuell sein kann, dass die Performance darunter leidet, wenn eine externe Datei jedesmal ausgelesen werden muss, habe ich den einfacheren Ansatz, die Parameter über setter methoden ändern zu können, ebenfalls implementiert. Will man dies nutzen, reicht eine kleine Änderung im Quellcode.
Das Frontend benutzt eine Liste von ausgewählten Städten. Auch diese Liste könnte man aus einer externen Date auslesen, allerdings könnte man dafür auch einen node-Service schreiben, eventuell einer, der Postleitzahlen auf Geodaten mappt.
Implentiert ist zur Zeit nur, was einfach war und schnell ging.
